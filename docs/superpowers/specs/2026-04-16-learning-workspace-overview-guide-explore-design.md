# 导学工作区 `Overview / Guide / Explore` 改版设计

**日期：** 2026-04-16

**目标：** 在保持当前 App 整体视觉语言与“导学”tab 首页不变的前提下，重做“进入某个导学本后的工作区”，让它能够承接新的 `Guide / Explore / Bridge` 后端能力，并形成一套更适合比赛展示、也更适合真实学习使用的页面结构。

## 背景

当前导学前端仍然是旧的 `tutor` 工作区心智：

- 入口层以“导学本库”为核心，用户点击某个导学本后直接进入旧的对话式 `Guide` 页面
- 工作区内部主要由 `guide / sources / more` 三个页面构成
- 页面重点仍然是“单导师式对话推进”

而后端已经升级为新的学习模型：

- `Guide Mode`：按步骤推进、评测、补救
- `Explore Mode`：NotebookLM 风格自由问答
- `Bridge Action`：在 `Guide` 与 `Explore` 间显式桥接

因此问题不再是“原页面做得够不够精细”，而是“当前页面结构已经不能准确表达后端能力”。

本轮设计的重点，是把工作区从“一个聊天页 + 两个附属页面”升级为“一套学习空间”，但同时不能破坏当前 App 已经建立起来的视觉气质和使用习惯。

## 用户确认后的边界

本轮边界已经明确，不做额外扩散：

- **保留不改：**
  - 当前 `导学` tab 首页
  - 当前 App 的整体视觉语言、版式气质、导航风格
- **本轮重点改版：**
  - 点击进入某个导学本之后的全部工作区页面
- **不作为本轮重点：**
  - 书籍详情页的触发入口重做
  - 导学 tab 首页重构
  - 完整 notebook 编辑器
  - 播客、文档 transformation、复杂工作室功能

## 方案选择

本轮采用用户确认的方案：

- 保留 `导学` tab 首页现状
- 点击导学本后，默认先进入一个新的 `Overview` 页面
- `Guide` 与 `Explore` 作为工作区内的两个并列核心模式
- `Bridge` 不是独立页面，而是两个模式间的关键动作

不采用“继续以 Guide 页为默认首页，再把 Explore 做成附属抽屉”的原因：

- 会弱化后端 `Guide / Explore / Bridge` 的完整升级
- 仍然会把工作区理解为“一个聊天页”
- 比赛展示时难以清晰讲出双模式与桥接能力

不采用“Guide / Explore 同屏分栏”的原因：

- 移动端信息密度过高
- 不符合当前 App 偏阅读、偏留白、偏分段式的信息组织方式
- 对日常学习使用不够友好

## 总体结构

本轮导学工作区的结构定义为：

1. `Overview`
2. `Guide`
3. `Explore`
4. `Sources & Graph`
5. `Review / Report`

这 5 个页面共同构成“导学本内部工作区”，替代当前工作区的旧三页结构。

### 核心原则

- `Overview` 负责分流和决策
- `Guide` 负责主线推进
- `Explore` 负责自由探索
- `Sources & Graph` 负责证据解释
- `Review / Report` 负责阶段收口

这意味着工作区不再是“一个主页面 + 两个边角补充页”，而是“两个核心学习模式 + 两个支撑页面 + 一个总览入口”。

## 路由与页面层级

建议的新工作区路由结构：

- `/tutor/[profileId]` -> `Overview`
- `/tutor/[profileId]/guide`
- `/tutor/[profileId]/explore`
- `/tutor/[profileId]/graph`
- `/tutor/[profileId]/review`

说明：

- 进入某个导学本时默认落到 `/tutor/[profileId]`
- 当前 `(workspace)/(search) / sources / more` 的文件结构应视为实现遗留，不再作为最终信息架构
- `info-sheet` 未来更适合作为轻量补充说明，不再承担主信息页职责

## 页面详细设计

### 1. `Overview`

**定位：** 导学本首页与模式分流中心。

**页面目标：**

- 让用户一眼知道当前导学本的状态
- 明确告诉用户“下一步最值得做什么”
- 在 `Guide` 与 `Explore` 之间做清晰分流

**核心区块：**

- 顶部状态卡
  - 导学本标题
  - 当前状态
  - 最近学习时间或当前步骤摘要
- 双主行动卡
  - `继续 Guide`
    - 当前步骤标题
    - 已完成步数
    - 最近一次评测结果
  - `进入 Explore`
    - 最近一次探索主题，或推荐首问
    - 若当前是从某一步展开过 Explore，需要给出焦点提示
- 今日建议区
  - 只保留 2 到 3 条高价值动作
  - 例如：继续当前步骤、回补某个概念、把一次 Explore 结果收编
- 辅助入口区
  - `资料与图谱`
  - `阶段复盘`

**设计原则：**

- 不是聊天入口页，而是学习入口页
- 不是功能目录，而是状态驱动页
- 卡片层级要清楚，避免“所有入口一样重要”

### 2. `Guide`

**定位：** 按步骤推进的主学习页面。

**页面目标：**

- 承接学习主线
- 清楚显示当前步骤、通过标准与推进结果
- 让评测与补救变成工作区内自然的一部分

**核心区块：**

- 当前步骤卡
  - 步骤标题
  - 目标
  - 通过标准
  - 当前进度状态
- 导学对话流
  - 保留现有聊天连续感
  - 但每轮都明确属于“当前步骤”
- 步骤反馈区
  - 达标时显示推进结果
  - 未达标时显示补救建议
- 快捷动作区
  - `展开自由探索`
  - `查看本步依据`
  - `返回总览`

**设计原则：**

- 主体仍然是学习，不是泛聊天
- 对话区要有“步骤容器感”，不能像普通聊天应用
- 补救与推进必须有强反馈，不可埋在消息细节里

### 3. `Explore`

**定位：** NotebookLM 风格的自由探索页。

**页面目标：**

- 支持围绕资料的自由提问
- 让答案、引用、概念成为同级信息
- 成为从主线学习岔出去深挖的空间

**核心区块：**

- 焦点卡
  - 若从 `Guide` 展开而来，显示当前聚焦的步骤
  - 若为纯自由探索，则显示“当前围绕整份资料自由提问”
- 自由问答流
  - 用户问题
  - Assistant 回答
  - 每轮挂接引用与相关概念
- 相关概念区
  - 可点击扩展，但本轮先按展示设计，不要求复杂交互
- 探索动作区
  - `收编到当前步骤`
  - `继续追问`
  - `查看概念图`
  - `返回总览`

**设计原则：**

- 不做成第二个 Guide
- 不做成只有问答文本的页
- 必须让“引用”和“相关概念”具备独立视觉地位

### 4. `Sources & Graph`

**定位：** 资料依据与知识关系页。

**页面目标：**

- 回答“这个答案从哪里来”
- 回答“这些概念之间怎么连”
- 强化比赛中的“可解释 AI”表达

**核心区块：**

- 来源资料区
  - 当前导学本使用的资料列表
  - 每份资料的摘要与来源类型
- 引用依据区
  - 当前步骤或当前 Explore 回答的主要证据
- 知识图谱区
  - 概念节点与关系图
  - 本轮只规划成展示型，不规划复杂编辑行为

**设计原则：**

- 不是“原 sources 页加一点图”
- 应作为可解释性的专属页面
- 页面语言偏证据和结构，而不是偏任务和对话

### 5. `Review / Report`

**定位：** 阶段学习结果页。

**页面目标：**

- 对当前学习阶段做结果收束
- 告诉用户“已经学会了什么、还缺什么、下一步去哪”
- 承接比赛展示中的结果输出

**核心区块：**

- 掌握度概览
- 已完成步骤
- 当前薄弱点
- 下一步建议
- 返回 `Guide` 或继续 `Explore` 的行动按钮

**设计原则：**

- 不做工具杂烩页
- 不延续当前 `more` 页“更多功能”的组织逻辑
- 只围绕学习结果和后续建议展开

## Bridge 设计

`Bridge` 在本轮中是关键动作，不是独立页面。

### `Guide -> Explore`

语义：

- 从当前导学步骤展开自由探索

表现：

- 在 `Guide` 页出现为明确动作按钮
- 进入 `Explore` 后应自动带上当前步骤上下文
- 页面顶部必须提示“当前探索围绕哪一步”

### `Explore -> Guide`

语义：

- 把自由探索结果收编进当前导学步骤

表现：

- 在 `Explore` 中以强操作按钮出现
- 收编成功后不强制跳回 `Guide`
- 允许用户：
  - 继续 Explore
  - 回到 Guide 继续主线

### 设计要求

- `Bridge` 必须是工作区最具 AI 感的动作之一
- 但不能把它做成复杂流程向导
- 它应该像“自然地在主线学习和发散探索间切换”

## 旧页面映射

### 当前 `guide`

拆为：

- `Overview`
- `Guide`

### 当前 `sources`

升级为：

- `Sources & Graph`

### 当前 `more`

重组为：

- `Review / Report`

剩余“工作室/占位型功能”不作为本轮页面主轴，应弱化或后移。

## 数据依赖与前后端契约映射

本轮页面规划直接建立在新的 `/api/v2/learning/*` 契约之上。

### `Overview`

至少需要聚合：

- profile
- 当前 active path version
- 当前 session 状态
- 最新 report
- 可进入的 `Guide / Explore` 入口状态

### `Guide`

主要依赖：

- `guide` session detail
- `POST /sessions/{id}/stream`
- 当前步骤、checkpoint、remediation、report

### `Explore`

主要依赖：

- `explore` session detail
- `POST /sessions/{id}/stream`
- turn 中的 citations、related concepts
- `bridge-actions`

### `Sources & Graph`

主要依赖：

- `GET /profiles/{id}/graph`
- profile sources
- 当前 turn 或当前步骤的 evidence 信息

### `Review / Report`

主要依赖：

- `GET /sessions/{id}/report`
- 当前 session / profile 的学习状态信息

## 状态设计

每个页面都必须显式覆盖以下状态，而不是只做 happy path。

### `Overview`

- 正在生成导学本
- 导学本可进入
- 导学本失败
- 没有活跃 session

### `Guide`

- 初次进入、尚未开始对话
- 正在流式返回
- 当前步骤已推进
- 当前步骤触发补救
- 已完成全部步骤

### `Explore`

- 无历史问答
- 有焦点上下文的探索
- 普通自由探索
- 收编成功反馈

### `Sources & Graph`

- 有图谱
- 图谱 fallback
- 仅有来源、图谱不可用

### `Review / Report`

- 有完整阶段结果
- 只有部分结果
- 仍在生成阶段报告

## 视觉与交互原则

### 保持与现有 App 风格一致

本轮明确不采用完全独立的视觉语言。

需要延续的现有特征：

- 编辑感、书房感、知识产品气质
- 柔和卡片与清晰分段
- 有插画感但不过度装饰
- 阅读优先而非控制台优先

### 页面语义需要更强

尽管视觉气质要保持一致，但工作区内部的信息结构需要更明确：

- `Overview` 看起来就应该像入口页
- `Guide` 看起来就应该像推进页
- `Explore` 看起来就应该像探索页
- `Review` 看起来就应该像结果页

不能只靠顶部标题区分页面。

## 风险与控制

- 风险：页面变多后，工作区显得更重
  - 控制：保留 `Overview` 作为唯一入口，不在一级层级堆过多标签
- 风险：`Guide / Explore` 视觉过于相似，用户分不清
  - 控制：通过页面结构、主操作和信息优先级拉开差异
- 风险：为了比赛展示而损失日常使用感
  - 控制：保持 App 原有视觉语言，不做夸张“AI 仪表盘化”
- 风险：旧组件迁移时出现概念混杂
  - 控制：先按新页面语义拆边界，再决定旧组件复用方式

## 测试与验收建议

前端实现时，至少需要覆盖以下验证：

### 页面级

- 进入导学本默认落到 `Overview`
- `Overview` 能正确跳转到 `Guide / Explore / Graph / Review`
- `Guide` 和 `Explore` 间跳转语义正确

### 行为级

- `Guide -> Explore` 能带上当前步骤焦点
- `Explore -> Guide` 的收编动作有明确反馈
- `Guide` 推进与补救状态在 UI 中有明显差异

### 回归级

- 不改动现有导学 tab 首页
- 不破坏现有 App 的导航与整体风格

## 交付顺序建议

1. 先重构工作区路由结构，建立 `Overview / Guide / Explore / Graph / Review`
2. 再完成 `Overview` 页面，因为它决定整个工作区入口心智
3. 然后分别重做 `Guide` 与 `Explore`
4. 最后完成 `Sources & Graph` 与 `Review / Report`
5. 收尾阶段再清理旧 `sources / more / info-sheet` 的职责

## 最终定义

本轮改版后的导学工作区，不再被定义为：

- “一个聊天页加两个附属页面”

而应被定义为：

- **一个以 `Overview` 为入口、以 `Guide / Explore` 为双核心、以 `Sources & Graph / Review & Report` 为支撑页的学习工作区。**
